# Trick 1: HTTP/1.1 and HTTP/2 Keep-Alive (Persistent TCP Connections)

While HTTP is _logically_ stateless,
the underlying **Layer 4 TCP connection does not have to close** after every request.

In old-school HTTP/1.0, connections did drop instantly.
But since HTTP/1.1,
every connection uses a default header called **`Connection: keep-alive`**.

### How it works

1. The browser runs the heavy Asymmetric TLS Handshake
   **exactly once** to open a secure TCP pipe to your server.

2. The browser requests `index.html`. The server responds.

3. Instead of tearing down the socket,
   **the TCP connection stays open** in a `Keep-Alive` state.

4. The browser immediately uses that _exact same,
   already-encrypted pipe_ to request `style.css`,
   then `app.js`, then your images.

Because the underlying pipe never closed,
the asymmetric math only happened **one time** at the very beginning.
The browser moves dozens of assets over that single secure pipeline
using the blazing-fast symmetric session key.

## Trick 2: What if the connection _does_ close? (TLS Session Resumption)

Eventually, the user leaves your site, or the idle timeout hits,
and the TCP connection finally closes.
10 minutes later, the user clicks another link and comes back.

Do they have to do the heavy asymmetric handshake all over again? **No.**

TLS has a built-in caching system called **Session Resumption**
(via _Session IDs_ or _Session Tickets_).
Think of it like a browser cookie, but strictly for cryptography.

### The "Session Ticket" Dance (TLS 1.3)

1. During the very first handshake,
   after doing the heavy asymmetric math,
   the server encrypts a tiny piece of state data called a
   **Session Ticket** using a secret key only the server knows,
   and hands it to the client.

2. The client saves this ticket in its memory cache.

3. When the client returns later and opens a brand new TCP connection,
   it sends a `ClientHello` and attaches that **Session Ticket**.

4. The server decrypts the ticket,
   says _"Ah, I remember you! We already calculated a safe secret key earlier,"_
   and **instantly skips the entire asymmetric public/private key phase.**

The server resumes the session immediately,
switching straight to fast symmetric encryption.
A full asymmetric handshake takes two round-trips over the network (2 RTT);
a resumed session or a modern TLS 1.3 connection
can drop that down to a fraction of the time (1 RTT or even 0 RTT).

---

## The Engineer's Summary

- **HTTP is stateless**
  at the application layer (it forgets who you are between requests).

- But **TCP is stateful**
  at the transport layer (it maintains a live, active socket stream pipe).

- **TLS hitches a ride on TCP persistence.**
  As long as the TCP connection is kept alive,
  or the server remembers your Session Ticket,
  you completely bypass the slow asymmetric math.
