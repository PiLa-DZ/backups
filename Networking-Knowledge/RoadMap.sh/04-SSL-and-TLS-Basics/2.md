## What actually happens in TLS: The "Server-Only" Key Swap

In a standard TLS handshake
(like when your browser visits a website or connects to your phone's SFTP server),
**only the server uses a Public/Private key pair.**
The client usually does _not_ have one.

Instead of swapping keys both ways,
the handshake uses a mix of asymmetric math and a quick mathematical calculation
to generate a **temporary, shared secret key**.

Here is the breakdown of the real dance:

### Step 1: The Server Sends Its Public Key

The server sends its **Digital Certificate** over to the client.
Inside that certificate is the server's **Public Key**.
(The server keeps its Private Key locked down tight inside its own memory).

### Step 2: The Client Invents a Secret Key

The client (your browser or app) verifies the certificate is real. Then,
instead of sending its own public key back,
the client creates a brand new, random, temporary piece of data.

### Step 3: The Client Encrypts it with the Server's Public Key

The client takes that temporary random data,
runs it through an encryption algorithm using the **Server’s Public Key**,
and sends the scrambled result back to the server.

- Because it was encrypted with the Server's Public Key,
  **absolutely no one on the internet can read it**—
  not even a hacker sniffing the Wi-Fi with Wireshark.

- Only the server can decrypt it using its matching **Private Key**.

### Step 4: Both Sides Calculate the Final Symmetric Key

Now, both the client and the server have this shared piece of random data.
They both plug it into a mathematical formula to generate the exact same
**Symmetric Session Key**.

From that point on,
they completely stop using public/private keys!
They use that single, shared symmetric key to encrypt
and decrypt the rest of your files and data.

---

## Why does it work this way instead of a two-way swap?

### 1. It is massively faster (Performance)

Asymmetric encryption (Public/Private keys) requires incredibly heavy mathematical processing numbers.
If a server had to do two-way public/private key calculations
for millions of users visiting a website every minute,
the server's CPU would melt.

Symmetric encryption (a single shared secret key)
is lightning-fast and requires almost zero CPU power.
TLS uses the public key for just a fraction of a millisecond
at the start to pass a secret, then switches to the faster method.

### 2. Browsers don't need identity verification

Think about it:
when you visit a bank website,
you need to be 100% sure the server is the actual bank
(so the server needs a public certificate to prove its identity).
But the bank doesn't care who your _browser_ is;
it just wants to open a secure pipe.
Once the secure pipe is open,
you prove who you are by typing your username and password
into a form _inside_ that encrypted tunnel.

## Where your theory IS 100% Right: SSH and Mutual TLS (mTLS)

Your idea of a two-way public/private key swap _is_ actually used in two specific scenarios:

1. **SSH Connections:**
   When you set up SSH keys on your laptop,
   you manually copy your laptop's public key onto your server.
   In that specific architecture, **both sides do have a key pair**,
   and they verify each other's public keys to log you in without a password.

2. **mTLS (Mutual TLS):**
   In highly secure backend microservices (like gRPC architectures),
   engineers configure **Mutual TLS**.
   Because server-to-server communication requires both backend machines
   to prove their identity to each other,
   both servers will swap public certificates exactly like you described!
